# 卡片生命周期管理 SOP

> 编码：KB_SOP_002_v1.0.0 | 版本：1.0.0 | 2026-06-27 | 状态：active
> 监管要求：IATF16949:2016 §7.5 成文信息控制

## 1. 目的

规范 MyBrain 知识库卡片的全生命周期管理——从创建到归档——确保每张卡片符合 IATF16949 §7.5 成文信息控制要求：
- 文件在受控状态下运行
- 各使用场所获得适用版本
- 误用失效/作废文件的风险消除

## 2. 范围

适用于 MyBrain/cards/ 下所有概念/方法/案例卡片的生命周期管理。

## 3. 职责

| 角色 | 职责 |
|------|------|
| 卡片创建者（各 Agent） | 按本 SOP 创建卡片，填写完整 frontmatter |
| CEO / 知识库管理员 | 定期审查卡片合规性，维护受控文件清单 |
| 审核者 | 评审卡片内容准确性，批准发布 |

## 4. 流程图

```
创建 ──→ 评审 ──→ 发布（受控） ──→ 使用 ──→ 修订评审 ──→ 归档/销毁
  │                    │                           │
  │                    ▼                           ▼
  │              分配编码 + 版本号              更新受控清单
  │              加入受控文件清单
  ▼
 如不通过 → 打回修改
```

## 5. 操作步骤

### 5.1 创建

**触发条件：** 新知识需要沉淀时（日常工作中遇到新概念、方法、案例）

**操作：**
1. 复制 `templates/卡片模板.md` 作为起始文件
2. 填写 frontmatter：

```yaml
---
title: {卡片标题}
type: 概念 | 方法 | 案例
code: KB_CAR_{类型}_{序号}_v1.0.0
version: 1.0.0
created: {YYYY-MM-DD}
updated: {YYYY-MM-DD}
tags: [{#标签}]
status: draft
---
```

3. 文件命名：`MyBrain/cards/{类型}/{主题}.md`
4. 内容撰写：按模板结构（摘要→详情→核心要点→相关）
5. 保存为 `draft` 状态，不纳入受控文件清单

**判定标准：**
- frontmatter 必填字段完整
- 内容有 `## 摘要`（一句话，不超过50字）
- 内容有 `## 相关`（至少1个 wikilink）
- 无敏感信息（脱敏检查通过）

### 5.2 评审

**触发条件：** 卡片草稿完成

**操作：**
1. 自查内容完整性（前序判定标准）
2. 提交审核（通过企业微信/飞书通知知识库管理员）
3. 审核内容：
   - 准确性：概念定义是否准确
   - 完整性：是否符合格式要求
   - 重复性：是否与现有卡片重复
   - 规范性：命名、frontmatter、wikilink 是否正确

**判定标准：**
- 通过审核 → 进入发布环节
- 不通过 → 打回创建者修改，状态保持 draft

### 5.3 发布（受控）

**触发条件：** 审核通过

**操作：**
1. 将 frontmatter 中的 `status: draft` 改为 `status: active`
2. 申请分配正式编码（如果尚未分配）
3. 更新 `system/受控文件清单.md`，新增条目
4. 在 `daily/YYYY-MM-DD.md` 中记录发布动作

**发布检查清单：**
- [ ] frontmatter 必填字段完整（title/type/code/version/created/updated/tags/status）
- [ ] 卡片命名符合规范（类型/主题）
- [ ] 已加入受控文件清单
- [ ] 已记录变更日志

### 5.4 使用

**触发条件：** 日常工作中需要参考或引用

**操作：**
1. 通过 `system/导航.md` 或受控文件清单检索
2. 引用时使用 wikilink 格式：`[[卡片名称]]`
3. 如发现卡片内容过时/有误 → 触发修订流程

### 5.5 修订

**触发条件：** 内容需要更新（发现错误、补充新信息、链接变化）

**操作：**
1. 填写变更申请（inbox/ 或直接在卡片内注释变更内容）
2. 识别变更范围：
   - patch（错别字修正）：直接修改，不换版本号
   - minor（内容增减 <20%）：minor +0.1（如 v1.0.0 → v1.1.0）
   - major（重写 ≥20%）：major +1（如 v1.1.0 → v2.0.0）
3. 更新 frontmatter：
   - `updated: {新日期}`
   - `version: {新版本号}`
4. 在卡片底部变更记录区记录变更历史
5. 如涉及受控文件清单 → 同步更新清单

**IATF16949 提醒：**
- 修订须保留变更历史（谁、何时、变什么）
- 重大修订须重新评审

### 5.6 归档/销毁

**触发条件：** 卡片内容已完全过时、被新卡片替代、或不再适用

**操作：**
1. 将 `status: active` 改为 `status: obsolete`
2. 文件名加 `_obsolete` 后缀
3. 移动至 `memory/archive/` 目录
4. 从受控文件清单中移除（注明移除日期和原因）
5. 在 `daily/YYYY-MM-DD.md` 中记录归档动作
6. 如有替代卡片 → 在替代卡片中更新相关链接

**作废文件保存期：1年**（参考 IATF16949 §7.5.3.4 作废文件控制要求）

## 7. 受控文件清单联动

每张 active 状态的卡片须在 `system/受控文件清单.md` 中登记，字段包括：
- 卡片名称
- 类型
- 编码
- 版本
- 创建日期
- 状态
- 最后评审日期
