# 敏感信息脱敏规范

> 编码：QL_SOP_002_v1.0.0 | 版本：1.0.0 | 2026-06-27 | 状态：active
> 监管要求：IATF16949:2016 §7.5 成文信息控制
> 适用范围：所有向 .md 文件写入信息的操作（memory/、MyBrain/、cron 输出等）

## 1. 目的

防止敏感信息（API Key、Token、凭证、个人信息）以明文形式写入知识库或日志文件，确保：
- 所有写入 .md 文件的信息经过脱敏检查
- 敏感凭证不被泄露到日志或知识库
- 泄露后有追溯机制

## 2. 敏感信息分类

| 类型 | 示例 | 脱敏规则 |
|------|------|---------|
| **API Key** | `sk-cp-abc123...xyz` | → `sk-cp-***` |
| **Bearer Token** | `Bearer eyJhbGciOi...` | → `Bearer ***` |
| **App Secret** | `abc123xyz`（飞书等） | → `***` |
| **密码** | 任何形式的登录密码 | → `***`（完全禁止写入） |
| **私钥/SSH Key** | `-----BEGIN RSA PRIVATE KEY-----` | → `***` |
| **身份证号** | 公民身份号码 | → `***-****-****-{后4位}` |
| **手机号** | 138-xxxx-xxxx | → `138-****-{后4位}` |
| **邮箱** | user@example.com | → `u***@e***.com` |
| **App ID** | `cli_a9f5...cd4` | 非敏感，可保留（仅标识符） |

## 3. 脱敏操作规则

### 3.1 强制检查时机

**每次**向 .md 文件写入信息前，必须执行脱敏检查：

1. **写入前扫描**：检查即将写入的内容
2. **匹配到敏感信息** → 立即脱敏后写入
3. **写入后确认**：读取确认写入内容已脱敏

### 3.2 脱敏操作表

| 类型 | 原始格式 | 脱敏后 |
|------|---------|--------|
| OpenAI API Key | `sk-abc123...xyz` | `sk-***` |
| MiniMax API Key | `sk-cp-abc123...xyz` | `sk-cp-***` |
| Feishu App Secret | `app_secret_xxx` | `***` |
| Bearer Token | `Bearer eyJhbGciOi...` | `Bearer ***` |
| 密码 | 任意密码字符 | `***`（完全不出现） |
| SSH 私钥 | `-----BEGIN...` | `***` |

### 3.3 脱敏示例

```python
# 正确示例
api_key = "sk-cp-abc123xyz789"
safe_key = "sk-cp-***"  # ✅ 脱敏后写入文件

# 错误示例
api_key = "sk-cp-abc123xyz789"
with open("file.md", "w") as f:
    f.write(f"API Key: {api_key}")  # ❌ 明文泄露

# Bearer Token
token = "Bearer eyJhbGciOi..."
safe = "Bearer ***"  # ✅
```

## 4. 扫描工具

### 4.1 手动扫描命令

```bash
# 扫描明文 API Key
grep -rE "sk-[a-zA-Z0-9]{20,}|sk-cp-[a-zA-Z0-9]{20,}" ~/.openclaw/workspace/MyBrain/ --include="*.md"

# 扫描 Bearer Token
grep -rE "Bearer [a-zA-Z0-9\-_]+\.[a-zA-Z0-9\-_]+\.[a-zA-Z0-9\-_]+" ~/.openclaw/ --include="*.md"

# 扫描密码关键词
grep -rE "password:|passwd:|密码:|pwd:" ~/.openclaw/ --include="*.md"
```

### 4.2 自动化扫描

Cron `weekly_quality_check` 每周自动扫描：
- 扫描 `memory/` 和 `MyBrain/` 下所有 .md 文件
- 发现敏感信息 → 立即触发 P1 告警 + 自动脱敏

## 5. 已发生的泄露处理

### 5.1 泄露发生时的步骤

1. **立即止血**：Revoke 受影响的凭证，生成新凭证
2. **评估影响**：是否已外泄、被利用
3. **通知相关方**：如影响飞书等外部服务 → 通知对方
4. **更新所有引用**：将旧凭证全部替换为新凭证
5. **记录泄露报告**：写入 `daily/事件报告_YYYY-MM-DD.md`

### 5.2 泄露报告内容

```markdown
## 泄露事件：{凭证类型}

> 发现时间：YYYY-MM-DD HH:MM
> 泄露位置：{文件路径}
> 凭证前半部分：{脱敏后的前缀}

### 影响评估
<!-- 是否有外泄风险 -->

### 处理措施
- [ ] Revoke 旧凭证
- [ ] 生成新凭证
- [ ] 更新所有引用点

### 预防措施
<!-- 如何避免再次发生 -->
```

## 6. 禁止事项

- **禁止**将任何形式的密码写入 .md 文件
- **禁止**将完整 Bearer Token 写入 .md 文件
- **禁止**将私钥写入 .md 文件
- **禁止**在 Cron 输出中包含真实凭证
- **禁止**在日志文件名中暴露凭证
- **禁止**通过企业微信/飞书发送凭证
