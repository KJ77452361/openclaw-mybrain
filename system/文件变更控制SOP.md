# 文件变更控制 SOP

> 编码：KB_SOP_003_v1.0.0 | 版本：1.0.0 | 2026-06-27 | 状态：active
> 监管要求：IATF16949:2016 §7.5.3.8 工程变更控制

## 1. 目的

规范 MyBrain 知识库中所有受控文件的变更管理，确保：
- 变更有明确的申请、评审、批准流程
- 变更后的文件版本明确，可追溯
- 相关方能及时获得最新有效版本
- 不会出现误用旧版本的情况

## 2. 范围

适用于 MyBrain 内所有受控文件，包括：
- system/ 下的所有标准文件（KB_STD_*）
- 所有 active 状态的卡片
- templates/ 下的模板文件
- 受控文件清单

## 3. 定义

| 术语 | 定义 |
|------|------|
| 受控文件 | status=active、已录入受控文件清单的文档 |
| 非受控文件 | status=draft 或含 `_draft` 的文件，仅供参考 |
| 作废文件 | status=obsolete 或含 `_obsolete`，已从受控清单移除 |
| ECN | Engineering Change Notice，工程变更通知 |
| 变更分级 | patch / minor / major（见版本规范） |

## 4. 职责

| 角色 | 职责 |
|------|------|
| 变更申请人 | 填写变更申请，记录变更内容和原因 |
| 审核者 | 评审变更影响，识别涉及文件，给出意见 |
| 批准者 | 批准或否决变更（CEO 或域负责人） |
| 执行者 | 实施变更，更新版本号，发布新文件 |
| 文控 | 更新受控文件清单，通知相关方 |

## 5. 变更分级标准

| 分级 | 触发条件 | 版本变化 | 评审要求 |
|------|---------|---------|---------|
| **patch** | 错别字修正、格式调整、描述优化 | v{x}.{y}.{z} → v{x}.{y}.{z+1} | 申请人自检即可 |
| **minor** | 内容增减 <20%（新增小节、补充案例） | v{x}.{y}.{z} → v{x}.{y+1}.{0} | 需相关方确认 |
| **major** | 重写 ≥20%、结构重组、重大方向调整 | v{x}.{y}.{z} → v{x+1}.{0}.{0} | 需正式评审 + 批准 |

## 6. 变更流程

### Phase 1：申请（Change Request）

```
触发条件：发现文件需要变更
执行动作：
  1. 变更申请人填写变更申请
  2. 说明：变更内容、原因、涉及文件
  3. 判定变更级别（patch/minor/major）
  4. 提交审核
```

**变更申请内容（记录在 daily/ 或 inbox/）：**

| 字段 | 内容 |
|------|------|
| 申请人 | 谁提出 |
| 申请日期 | YYYY-MM-DD |
| 目标文件 | 要修改的文件名和路径 |
| 当前版本 | 变更前的版本号 |
| 变更内容 | 具体修改了什么 |
| 变更原因 | 为什么要改 |
| 变更级别 | patch / minor / major |
| 影响文件 | 哪些相关文件也需要同步更新 |

### Phase 2：评审（Review）

```
触发条件：变更申请提交后
执行动作：
  1. 文控/管理员接收变更申请
  2. 评审变更影响范围：
     - 是否有相关文件需要同步更新？
     - 是否存在重复或冲突的内容？
     - 对现有工作流有何影响？
  3. minor 及以上：通知相关方征求意见
  4. 综合评审意见，给出结论：通过 / 否决 / 需修改
```

### Phase 3：批准（Approval）

| 变更级别 | 批准权限 |
|---------|---------|
| patch | 申请人自检通过即可 |
| minor | 域负责人批准 |
| major | CEO 批准 |

```
执行动作：
  1. patch：直接进入执行阶段
  2. minor：域负责人审批，记录批准结果
  3. major：CEO 审批，记录批准结果
  4. 如否决：变更申请人接收反馈，修改后重新申请
```

### Phase 4：执行（Implementation）

```
触发条件：变更批准后
执行动作（变更执行者）：
  1. 更新文件 frontmatter：
     - version → 新版本号
     - updated → 执行日期
     - 如有变更记录区，同步追加变更历史
  2. 文件名中的版本号同步更新
  3. 如涉及受控文件清单 → 同步更新清单
  4. 将变更记录写入 daily/YYYY-MM-DD.md
```

**执行检查清单：**
- [ ] 文件版本号已更新
- [ ] updated 日期已更新
- [ ] 变更历史已记录（在卡片末尾变更记录区）
- [ ] 相关文件已同步更新（如有）
- [ ] 受控文件清单已更新（如文件在其中）

### Phase 5：分发与回收（Distribution & Retrieval）

```
触发条件：新版本文件生效
执行动作（文控）：
  1. 在 daily/ 或通知渠道发布变更通知
     内容：文件名、变更摘要、变更级别、新版本号
  2. 通知相关方获取新版本
  3. 回收旧版本文件：
     - 旧版本加 `_obsolete` 后缀
     - 如为受控文件，从原位置移除
     - 旧版本保存至 memory/archive/
  4. 确认所有使用场所已更新
```

### Phase 6：归档（Archival）

```
执行动作：
  1. 旧版本文件：
     - 文件名加 `_obsolete_{日期}` 后缀
     - 移入 memory/archive/
     - 保存 1 年后评估销毁
  2. 变更记录：
     - 完整变更记录保留在 daily/YYYY-MM-DD.md
     - 作为受控文件清单的变更凭证
```

## 7. 紧急变更流程

如发生紧急情况（如安全漏洞、严重错误）需要立即变更：

1. 口头通知 CEO（或域负责人）
2. 立即执行修复（patch 级别即可）
3. 事后补填变更申请（24小时内）
4. 正常走后续评审和归档流程

---

## 8. 版本号更新规则

文件每发生一次实质性变更，版本号必须更新：

```
初始发布：v1.0.0
第1次 patch：v1.0.1
第2次 patch：v1.0.2
第1次 minor：v1.1.0（清零 patch 位）
第1次 major：v2.0.0（清零 minor 和 patch 位）
```

---

## 9. 变更记录要求

每张卡片/每个受控文件必须保留变更记录区（Change Log）：

```markdown
## 变更记录

| 日期 | 版本 | 变更内容 | 变更人 |
|------|------|---------|--------|
| 2026-06-27 | v1.0.0 | 首版发布 | King |
| 2026-07-01 | v1.0.1 | 修正错别字"自进化"→"自进化" | King |
```

---

## 10. 禁止事项

- **禁止**在未走变更流程的情况下直接修改 active 状态文件（patch 除外）
- **禁止**删除变更历史记录
- **禁止**在作废文件上继续编辑后重新激活（应创建新版）
- **禁止**在 active 和 draft 状态之间随意切换（必须走审批）
