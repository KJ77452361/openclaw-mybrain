# 自进化评估工作流

> 编码：KB_WF_005_v1.0.0 | 版本：1.0.0 | 2026-06-27 | 状态：active
> 监管要求：IATF16949:2016 §7.5 成文信息控制
> 适用范围：OpenClaw 生态自进化效果的定期评估与反馈闭环

## 1. 目的

建立自进化效果的量化评估体系，确保：
- 自进化的进展可通过客观指标衡量
- 每个进化循环的效果可验证
- 进化策略基于数据而非主观判断
- 评估结果驱动下一轮进化（形成闭环）

## 2. 评估周期

| 周期 | 评估内容 | 执行时间 | 执行者 |
|------|---------|---------|--------|
| 周评估 | 周内进化数据汇总 | 每周一 09:00 | CEO |
| 月评估 | 月度进化效果报告 | 每月最后一日 | CEO |
| 季评估 | 季度体系健康度审查 | 每季度末 | CEO + 域负责人 |

## 3. 评估指标体系

### 3.1 知识积累指标（Knowledge Accumulation）

| 指标 | 计算方式 | 目标 | 告警阈值 |
|------|---------|------|---------|
| 卡片新增率 | 新增卡片数 / 基准卡数 | 月增长 ≥5% | <2% |
| SOP 覆盖率 | 已有 SOP 的场景数 / 总场景数 | ≥80% | <60% |
| 知识复用率 | 被引用卡片数 / 总卡片数 | ≥50% | <30% |
| 模板使用率 | 使用模板的文档数 / 总文档数 | ≥70% | <50% |

### 3.2 自进化过程指标（Evolution Process）

| 指标 | 计算方式 | 目标 | 告警阈值 |
|------|---------|------|---------|
| Dreaming 执行率 | 实际执行次数 / 计划执行次数 | 100% | <80% |
| 洞察转化率 | 进入知识库的洞察数 / 总洞察数 | ≥30% | <15% |
| 进化闭环率 | 完成效果评估的变更数 / 总变更数 | ≥70% | <50% |
| 平均进化周期 | 从识别到应用的时间（天） | ≤7 天 | >14 天 |

### 3.3 系统健康指标（System Health）

| 指标 | 计算方式 | 目标 | 告警阈值 |
|------|---------|------|---------|
| 受控文件合规率 | 合规文件数 / 总文件数 | 100% | <95% |
| frontmatter 完整率 | 有 frontmatter 的文件 / 总文件数 | 100% | <90% |
| 敏感信息泄露数 | 扫描发现的明文凭证数 | 0 | ≥1 |
| Agent 响应正确率 | 正确响应次数 / 总响应次数 | ≥85% | <70% |

## 4. 评估流程

### Phase 1：数据收集

**数据来源：**

| 指标 | 数据来源 | 计算周期 |
|------|---------|---------|
| 卡片新增率 | `system/受控文件清单.md` | 月 |
| SOP 覆盖率 | `system/SOP索引.md` | 季 |
| Dreaming 执行率 | `memory/dreams/deep/` 文件计数 | 周 |
| 洞察转化率 | `memory/dreams/deep/` + `system/受控文件清单.md` | 月 |
| 受控文件合规率 | frontmatter 扫描脚本 | 月 |
| 敏感信息泄露数 | grep 扫描 | 周 |

**操作：**
1. 读取各数据来源
2. 计算各项指标
3. 与基准值/目标值对比
4. 识别异常项（超标/未达标）

### Phase 2：效果分析

**分析内容：**

| 分析项 | 说明 |
|--------|------|
| 达标项 | 哪些指标达到或超过目标？ |
| 异常项 | 哪些指标触发告警阈值？ |
| 趋势分析 | 指标是改善还是恶化？ |
| 根因分析 | 异常项的根本原因是什么？ |

**趋势判断标准：**
- 连续 3 周持平 → 需要结构性改变
- 连续 2 周恶化 → 触发紧急评估
- 单周大幅改善（>20%）→ 分析原因，看是否可复用

### Phase 3：策略调整

**基于分析结果，调整进化策略：**

| 分析结论 | 调整动作 |
|---------|---------|
| 某域进化过快 | 设置冷却期，避免资源透支 |
| 某域进化不足 | 识别瓶颈，增加该域的 Dreaming 关注度 |
| 指标全面恶化 | 触发系统性问题审查（KB_SOP_007） |
| 某 SOP 无效 | 进入修订或废弃流程（KB_SOP_003） |

### Phase 4：报告生成

**输出文件：**

| 周期 | 文件 | 保存位置 |
|------|------|---------|
| 周评估 | `daily/周评估_YYYY-WXX.md` | `daily/` |
| 月评估 | `daily/月评估_YYYY-MM.md` | `daily/` |
| 季评估 | `daily/季评估_YYYY-QX.md` | `daily/` |

**报告结构：**

```markdown
# 自进化周评估 — YYYY-WXX

> 评估周期：YYYY-MM-DD ~ YYYY-MM-DD
> 评估者：CEO

## 指标汇总

| 指标 | 目标 | 实际 | 状态 |
|------|------|------|------|
| 卡片新增率 | ≥5% | X% | ✅/⚠️/❌ |
| Dreaming 执行率 | 100% | X% | ✅/⚠️/❌ |
| 受控文件合规率 | 100% | X% | ✅/⚠️/❌ |
| ... | ... | ... | ... |

## 异常项分析
<!-- 对每个 ⚠️/❌ 项，分析原因 -->

## 策略调整
<!-- 基于分析的调整建议 -->

## 下周进化计划
- [ ] ...
```

### Phase 5：反馈闭环

**将评估结果反馈到进化循环：**

1. 异常项 → 触发 `inbox/问题单`
2. 策略调整 → 更新次周 Dreaming 关注方向
3. 无效 SOP → 触发 KB_SOP_003 修订流程
4. 系统性问题 → 触发 KB_SOP_007 告警流程

## 5. 评估责任矩阵

| 角色 | 周评估 | 月评估 | 季评估 |
|------|--------|--------|--------|
| CEO | 主导 | 主导 | 主导 |
| 域负责人（CTO/COO） | 配合 | 提供数据 | 共同评审 |
| 知识库管理员 | 提供数据 | 数据汇总 | 文档整理 |

## 6. 禁止事项

- **禁止**在数据不完整时出具评估报告（数据缺失需标注）
- **禁止**调整目标值来迎合未达标的结果
- **禁止**忽略连续 2 周以上的恶化趋势
- **禁止**评估报告与实际数据不符（数字必须可验证）
