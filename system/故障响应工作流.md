# 故障响应工作流

> 编码：KB_WF_002_v1.0.0 | 版本：1.0.0 | 2026-06-27 | 状态：active
> 监管要求：IATF16949:2016 §10.2 不合格与纠正措施
> 适用范围：OpenClaw 生态内所有系统异常、故障、失效事件

## 1. 目的

建立标准化的故障响应流程，确保：
- 故障能在最短时间内被识别和止血
- 升级路径清晰，责任到位
- 故障处理过程有记录，事后可复盘
- 防止同类故障重复发生

## 2. 故障分级

同 KB_SOP_007 §2（告警与升级路径 SOP）。

## 3. 故障响应阶段

```
检测 ──→ 确认 ──→ 止血 ──→ 修复 ──→ 复盘
```

## 4. 各阶段操作

### Phase 1：检测

**触发来源：**
- Cron 健康检查（`sys_health_daily`）每日 09:00 执行
- Heartbeat 定期检查
- 用户主动报告
- 子 Agent 运行异常

**操作：**
1. 确认故障存在（非误报）
2. 记录检测时间
3. 初判故障级别（P0-P4）
4. 如为 P0/P1 → 立即进入止血阶段

### Phase 2：确认

**操作：**
1. 收集故障证据（日志、错误信息、返回数据）
2. 确认影响范围（哪些功能/用户受影响）
3. 确认故障开始时间（最早发现时间）
4. 记录故障ID：`INC-{YYYYMMDD}-{序号}`
5. 通知相关方（按升级路径）

### Phase 3：止血（Containment）

**目标：** 防止故障扩大，不求根治，只求隔离

| 故障类型 | 止血措施 |
|---------|---------|
| 服务不可用 | 切换备用通道 / 回退到稳定版本 |
| 数据泄露 | 立即隔离相关凭证，Revoke 受影响的 Key |
| 外部服务中断 | 切换到备用 API / 降级功能 |
| 文件损坏 | 恢复备份 / 从受控文件清单重建 |

**操作：**
1. 执行止血措施
2. 确认止血生效（影响不再扩大）
3. 记录止血完成时间

### Phase 4：修复（Resolution）

**操作：**
1. 分析根本原因（RCA）
2. 制定修复方案
3. 执行修复（如需变更文件 → 走文件变更 SOP）
4. 验证修复有效
5. 确认恢复正常服务
6. 记录恢复时间

### Phase 5：复盘（Post-mortem）

**操作：**
1. 编写事件报告（按 KB_SOP_007 §8 模板）
2. 识别根本原因（RCA）
3. 制定预防措施（PA）
4. 将 PA 加入工作清单
5. 更新 SOP 或工作流（如流程本身有问题）
6. 关闭事件

## 5. 关键时间节点

| 节点 | 要求 |
|------|------|
| P0 检测→CEO知晓 | ≤5 分钟 |
| P1 检测→初步响应 | ≤30 分钟 |
| P2 检测→开始修复 | ≤2 小时 |
| 事件报告初稿 | 事件关闭后 24 小时内 |

## 6. 故障止血手册

### 6.1 Feishu 消息发送失败

**症状：** 消息发送返回 `failureAlert`

**止血：**
1. 检查 OpenClaw Gateway 连接状态
2. 检查 Feishu app 配置（appId/appSecret）
3. 确认飞书 bot 是否被禁言
4. 降级：切换到 webchat 通知

### 6.2 MiniMax API 超载（429）

**症状：** API 返回 429 或超载错误

**止血：**
1. 减少并发请求
2. 等待 Retry-After 时间
3. 如持续超载 → 触发容量告警

### 6.3 文件系统故障

**症状：** frontmatter 丢失、文件损坏

**止血：**
1. 从 git 历史恢复（如有版本控制）
2. 从受控文件清单重建
3. 隔离损坏文件到 `memory/archive/`

### 6.4 敏感信息泄露

**症状：** grep 扫描发现明文凭证

**止血（立即）：**
1. 确认泄露范围
2. Revoke 受影响的凭证
3. 更新飞书管理员（如果涉及飞书凭证）
4. 评估是否需要通知用户
