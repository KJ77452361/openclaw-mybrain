# Dreaming System 工作流

> 编码：KB_WF_004_v1.0.0 | 版本：1.0.0 | 2026-06-27 | 状态：active
> 监管要求：IATF16949:2016 §7.5 成文信息控制
> 适用范围：Dreaming System（夜间记忆整合与洞察生成）的标准执行流程

## 1. 目的

规范 Dreaming System 的执行流程——将每日 Session 记忆整合、提炼洞察、生成知识沉淀建议——确保：
- 每晚自动执行（或手动触发）
- 输出结构化的洞察报告
- 识别出的知识入库建议进入正式流程
- 执行结果可追溯

## 2. 触发机制

| 触发类型 | 触发时间 | 执行方式 |
|---------|---------|---------|
| 自动触发 | 每日 02:00（GMT+8） | Cron Job `dreaming_promotion` |
| 手动触发 | 任意时间 | CEO 发送心跳触发 |

## 3. 工作流阶段

```
输入收集 ──→ 记忆整合 ──→ 模式识别 ──→ 洞察提炼 ──→ 报告生成 ──→ 知识路由
```

### Stage 1：输入收集

**数据来源：**
- `memory/sessions/` — 过去 24 小时的 Session 记录
- `memory/daily/` — 过去 24 小时的每日记录
- `inbox/` — 待处理问题单

**收集操作：**
1. 确定 Dreaming 时间窗口（昨日 02:00 → 今日 02:00）
2. 读取窗口内所有 Session 文件
3. 读取 `daily/YYYY-MM-DD.md`
4. 读取 `inbox/` 中新增条目

**输出：** 原始数据集合（待整合）

### Stage 2：记忆整合

**操作：**
1. 按时间顺序排列所有事件
2. 识别任务完成链（任务A → 结果A → 任务B → 结果B）
3. 识别错误发生链（触发条件 → 错误 → 处理方式 → 结果）
4. 按主题聚类（同一类操作归为一组）

**整合输出：**
- 成功路径清单（高频成功模式）
- 失败模式清单（高频失败模式）
- 效率数据（Token 消耗、响应时间分布）

### Stage 3：模式识别

**分析内容：**

| 分析项 | 说明 |
|--------|------|
| 高频成功 | 出现次数 ≥3 的成功操作模式 |
| 高频失败 | 出现次数 ≥2 的失败模式 |
| 效率趋势 | Token 消耗是增长/持平/下降 |
| 用户意图 | 用户最常请求的任务类型 |
| 知识缺口 | 发现但未沉淀的重要经验 |

**判定标准：**
- 成功模式：同一操作路径出现 ≥3 次
- 失败模式：同一错误出现 ≥2 次
- 效率异常：单任务 Token 消耗超过均值 2 倍

### Stage 4：洞察提炼

**对每个识别出的模式，生成洞察：**

| 洞察类型 | 提炼内容 |
|---------|---------|
| 方法论洞察 | 成功路径可以固化为 SOP 吗？ |
| 问题洞察 | 失败模式有系统原因吗？能通过规则避免吗？ |
| 效率洞察 | 消耗异常的原因是什么？ |
| 知识缺口 | 哪些领域的知识还没有卡片？ |

**洞察质量标准：**
- 每条洞察须有「核心发现」+「建议行动」
- 建议行动分两类：立即行动（24h）/ 计划行动（周度）
- 无实质内容的观察不计入洞察（如"用户问了问题"）

### Stage 5：报告生成

**输出文件：** `memory/dreams/deep/YYYY-MM-DD.md`

**报告结构：**

```markdown
# Dreaming Report — YYYY-MM-DD

> 执行时间：YYYY-MM-DD HH:MM
> 输入来源：sessions/ + daily/ + inbox/
> 洞察数量：N 条

## 执行摘要
<!-- 一句话说本次 Dreaming 的核心结论 -->

## 成功模式
| 模式 | 出现次数 | 建议 |
|------|---------|------|
| ... | ... | ... |

## 失败模式
| 模式 | 出现次数 | 根因假设 | 建议 |
|------|---------|---------|------|
| ... | ... | ... | ... |

## 效率数据
<!-- Token 消耗、响应时间等 -->

## 知识缺口
| 领域 | 缺口描述 | 建议动作 |
|------|---------|---------|
| ... | ... | ... |

## 建议立即行动（24h）
- [ ] ...

## 建议周度计划
- [ ] ...

## 知识入库建议
| 主题 | 类型 | 优先级 |
|------|------|--------|
| ... | ... | 高/中/低 |
```

### Stage 6：知识路由

**对每条知识入库建议：**

| 建议类型 | 路由目标 |
|---------|---------|
| 新 SOP | `inbox/知识入库申请` → 触发 KB_SOP_006 |
| 新概念卡 | `inbox/知识入库申请` → 触发 KB_SOP_006 |
| 现有卡片更新 | 直接更新（走 KB_SOP_003） |
| 系统性问题 | 触发 KB_SOP_007（告警） |

## 4. 执行检查清单

- [ ] 输入数据已完整收集（sessions/ + daily/ + inbox/）
- [ ] 成功/失败模式已识别并量化
- [ ] 每条洞察有「核心发现」+「建议行动」
- [ ] 报告已生成并存入 `memory/dreams/deep/`
- [ ] 知识入库建议已路由到对应流程
- [ ] 建议立即行动已加入 `inbox/待办`

## 5. 已知问题处理

| 问题 | 处理方式 |
|------|---------|
| 输入数据为空 | 生成「无数据报告」，记录检查结果 |
| Session 文件损坏 | 跳过损坏文件，记录损坏文件名 |
| 无可提炼洞察 | 生成「无洞察报告」，说明原因 |
