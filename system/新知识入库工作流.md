# 新知识入库工作流

> 编码：KB_WF_003_v1.0.0 | 版本：1.0.0 | 2026-06-27 | 状态：active
> 监管要求：IATF16949:2016 §7.5 成文信息控制
> 适用范围：外部知识（网页、文档）向 MyBrain 知识库沉淀的全流程

## 1. 目的

建立从"发现外部有用知识"到"成为 MyBrain 受控卡片"的标准路径，确保：
- 每条入库知识有可验证的来源
- 经过评估才入库（无重复、无杂质）
- 全程可追溯（谁、何时、来自哪里）
- 入库后符合受控文件标准

## 2. 知识来源分类

| 来源类型 | 示例 | 处理路径 |
|---------|------|---------|
| Web 文章 | 技术博客、官方文档 | web_fetch → 评估 → 入库 |
| 内部经验 | 工作中的方法论 | 直接入库申请 |
| 对话产生 | 与用户的讨论中产生 | 提炼 → 入库申请 |
| Cron 输出 | 定期报告中的洞察 | 提炼 → 入库申请 |
| 外部文档 | PDF、PPT | 内容提炼 → 入库 |

## 3. 工作流阶段

```
发现 ──→ 抓取/提炼 ──→ 评估 ──→ 编制 ──→ 评审 ──→ 发布 ──→ 归档
```

### Stage 1：发现（Discovery）

**触发：** 发现有价值的外部知识

**操作：**
1. 保存原始 URL 到 `inbox/来源记录.md`
   ```
   - URL: https://example.com/article
   - 发现时间: 2026-06-27
   - 发现者: CEO
   - 摘要: 关于XXX的关键洞见
   ```
2. 在 `inbox/` 创建知识入库申请：`知识入库申请_YYYY-MM-DD.md`

### Stage 2：抓取/提炼（Capture）

| 来源 | 工具 | 操作 |
|------|------|------|
| Web 页面 | `web_fetch` | 提取 markdown/text 内容 |
| 内部经验 | 直接提炼 | 写入申请文档 |
| Cron 报告 | 内容提炼 | 提炼关键洞察 |

**操作：**
1. 对原始内容进行提炼（保留核心，删除冗余）
2. 提炼原则：
   - 概念类：保留定义 + 核心要点
   - 方法类：保留步骤 + 判定标准
   - 案例类：保留背景 + 经过 + 结果
3. 写入申请文档的 `## 内容摘要` 章节

### Stage 3：评估（Evaluation）

**执行者：** CEO 或知识库管理员

| 评估项 | 标准 | 判定 |
|--------|------|------|
| 准确性 | 内容经过验证，非臆测 | ✅/❌ |
| 适用性 | 对 OpenClaw 生态有参考价值 | ✅/❌ |
| 时效性 | 内容是最新的（非过时） | ✅/❌ |
| 完整性 | 内容是否足够丰富（≥3个要点） | ✅/❌ |
| 重复性 | 是否与现有卡片重复（>30%相似→合并） | 是/否 |
| 版权 | 是否有版权限制（如有 → 不入库） | ✅/❌ |

**决策：**
- 全部 ✅ + 无重复 → 进入编制
- 有 ❌ → 退回发现者，说明原因
- 重复 → 识别目标合并卡片，进入合并流程

### Stage 4：编制（Authoring）

**执行者：** 发现者或入库执行者

1. 确认知识类型（概念/方法/案例）
2. 分配 KB_CAR 编码（查 SOP索引.md）
3. 按 `卡片模板.md` 创建卡片
4. frontmatter：
   ```yaml
   code: KB_CAR_{类型}_{序号}_v1.0.0
   version: 1.0.0
   status: draft
   source:
     url: https://...   # 外部来源 URL（如有）
     discovered_by: CEO
     discovered_at: 2026-06-27
   ```
5. `## 来源` 章节必须填写（如为外部知识）
6. 完成后提交评审

### Stage 5：评审（Review）

**执行者：** CEO 或知识库管理员

| 检查项 | 标准 |
|--------|------|
| 格式 | frontmatter 完整，结构符合模板 |
| 来源 | `## 来源` 章节存在且准确 |
| 内容 | 无抄袭，原创或已注明来源 |
| 重复 | 与现有卡片内容重复度 <30% |
| 敏感 | 无敏感信息 |

**通过标准：** 全部检查项通过

### Stage 6：发布（Publication）

**执行者：** 入库执行者

1. `status: draft` → `status: active`
2. 更新 `system/受控文件清单.md`
3. 更新 `system/SOP索引.md`（如是新 SOP）
4. 在 `daily/YYYY-MM-DD.md` 记录：
   ```
   知识入库：{卡片名称}（KB_CAR_{类型}_{序号}）
   来源：{URL 或"内部产生"}
   ```

### Stage 7：归档（Archival）

**执行者：** 入库执行者

1. `inbox/来源记录.md` 更新状态为 `已入库`
2. `inbox/知识入库申请_YYYY-MM-DD.md` 移入 `inbox/processed/`
3. 如外部来源内容特别重要 → 原文备份至 `MyBrain/external/`

## 4. 快速路径（内部经验）

对于直接在工作中产生的知识（无需 web_fetch）：

```
识别 → 提炼 → 评估 → 编制 → 评审 → 发布
```

跳过 Stage 1（发现）和 Stage 2（抓取），直接提炼后进入评估。

## 5. 记录要求

| 记录 | 保存位置 | 保存期限 |
|------|---------|---------|
| 来源记录 | `inbox/来源记录.md` | 1 年 |
| 入库申请 | `inbox/知识入库申请_YYYY-MM-DD.md` | 处理后 30 天 |
| 已入库卡片 | `cards/{类型}/` | 永久 |
