# 每周复盘 SOP

> 编码：KB_SOP_010_v1.0.0 | 版本：1.0.0 | 2026-06-27 | 状态：active
> 监管要求：IATF16949:2016 §7.5 成文信息控制
> 适用范围：CEO 及各 Agent 域负责人的每周复盘

## 1. 目的

在每日复盘的基础上，以周为周期进行深度复盘，确保：
- 识别跨日模式（高频问题、效率趋势）
- 调整下周计划（基于本周数据）
- 触发 SOP 改进（如流程存在问题）
- 为月/季复盘提供数据聚合

## 2. 触发时机

| 类型 | 时间 | 执行者 |
|------|------|--------|
| 常规周复盘 | 每周五 21:00（或周末） | CEO |
| 紧急复盘 | 重大事件后 24h 内 | 相关 Agent |

## 3. 复盘流程

```
收集数据（daily/本周） → 五维度分析 → 模式识别 → 行动项 → 下周计划
```

### Phase 1：数据收集

| 数据来源 | 内容 | 读取位置 |
|---------|------|---------|
| 每日笔记 | 本周 7 份 daily/YYYY-MM-DD.md | daily/ |
| 问题单 | 本周所有 inbox/问题单 | inbox/ |
| Cron 报告 | 本周 Cron 执行记录 | daily/cron_* |
| 知识库变更 | 本周新增/修订卡片 | system/受控文件清单.md |

### Phase 2：五维度分析

使用 `templates/周复盘模板.md`（KB_WI_007）：

| 维度 | 问题 | 产出 |
|------|------|------|
| **目标回顾** | 本周原定目标是什么？完成了多少？ | 完成率 % |
| **任务完成** | 7 天共完成多少任务？哪些完成了？ | 任务清单 |
| **问题模式** | 哪些问题出现了多次？根因是什么？ | 问题模式分析 |
| **效率评估** | Token 消耗、响应时间与上周比如何？ | 效率趋势 |
| **知识沉淀** | 本周新增了多少卡片/洞察？ | 知识积累量 |

### Phase 3：模式识别

**重点识别：**
- 同一问题出现 ≥2 次 → 需要系统性解决
- 任务完成率 <70% → 需要调整工作方式
- 效率下降 >20% → 需要流程优化
- 无知识沉淀 → 需要加强复盘深度

**模式分类：**

| 类型 | 标记 | 处理方式 |
|------|------|---------|
| 偶发 | 🔵 偶发 | 记录，不触发跟进 |
| 趋势 | 🟡 趋势 | 加入下周关注 |
| 系统性 | 🔴 系统性 | 触发 KB_SOP_008（自进化）|

### Phase 4：行动项生成

| 优先级 | 来源 | 处理 |
|--------|------|------|
| P0 | 系统性问题（≥3次） | 立即触发自进化流程 |
| P1 | 趋势问题（2次） | 制定解决方案，纳入下周计划 |
| P2 | 优化项 | 记录到优化清单，择机处理 |

**行动项格式：**
```markdown
- [ ] {具体行动} @负责Agent 截止日期
```

### Phase 5：下周计划

| 维度 | 内容 |
|------|------|
| 核心目标 | 1-3 个最重要的目标 |
| 关键任务 | 5-7 个关键任务 |
| 关注风险 | 已识别风险及监控措施 |

## 4. 输出规范

| 输出 | 保存位置 | 说明 |
|------|---------|------|
| 周复盘报告 | `daily/周报_YYYY-WXX.md` | 本周复盘（永久） |
| 下周计划 | 合并到周报末尾 | 独立可执行 |
| 问题清单更新 | `inbox/问题单` | 跟踪未解决问题 |
