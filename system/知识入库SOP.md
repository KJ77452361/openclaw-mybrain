# 知识入库 SOP

> 编码：KB_SOP_006_v1.0.0 | 版本：1.0.0 | 2026-06-27 | 状态：active
> 监管要求：IATF16949:2016 §7.5 成文信息控制
> 适用范围：所有外部知识、内部经验向 MyBrain 沉淀的操作

## 1. 目的

规范从外部来源（web_search、文档、对话）或内部经验向 MyBrain 沉淀知识的流程，确保：
- 知识有明确的来源和上下文
- 知识经过评估后才进入正式知识库
- 知识卡片格式统一、可检索
- 重复知识被识别和合并

## 2. 适用范围

- 外部文档/文章关键内容沉淀
- 工作中产生的经验/教训记录
- Agent 协作产生的通用知识沉淀
- Cron 输出的定期总结入知识库

## 3. 职责

| 角色 | 职责 |
|------|------|
| 发现者 | 识别有价值的知识，提交入库申请 |
| 评估者 | 评估知识的准确性、适用性、重复性 |
| 审批者 | 批准知识入库（或打回） |
| 入库执行者 | 按本 SOP 创建卡片，更新清单 |

## 4. 知识分类与目标目录

| 来源 | 类型 | 目标目录 |
|------|------|---------|
| 外部文档/文章 | 概念 | `cards/概念/` |
| 操作经验/SOP | 方法 | `cards/方法/` |
| 项目复盘/案例 | 案例 | `cards/案例/` |
| 模板类 | WI | `templates/` |
| 系统性文档 | 制度/SOP | `system/` |

## 5. 入库流程

```
识别 ──→ 评估 ──→ 分类 ──→ 编制 ──→ 评审 ──→ 发布 ──→ 归档
```

### Phase 1：识别（Discovery）

**触发条件：** 发现有价值的知识

| 来源 | 发现时机 |
|------|---------|
| 外部文档 | web_search/web_fetch 后评估有用 |
| 内部经验 | 工作中产生洞见 |
| 协作产出 | Agent 协作产生可复用结论 |
| Cron 输出 | 定期报告中有价值的洞察 |

**操作：**
1. 将知识源/摘要写入 `inbox/YYYY-MM-DD_知识入库.md`
2. 填写来源 URL 或来源上下文（如有）
3. 标记知识类型（概念/方法/案例）

### Phase 2：评估（Evaluation）

**执行者：** 评估者（知识库管理员或 CEO）

| 评估项 | 标准 | 判定 |
|--------|------|------|
| 准确性 | 知识内容经过验证 | ✅/❌ |
| 适用性 | 对 OpenClaw 生态有参考价值 | ✅/❌ |
| 重复性 | 是否与现有卡片重复 | 是/否 |
| 时效性 | 是否为最新信息 | ✅/❌ |
| 完整性 | 内容是否足够丰富 | ✅/❌ |

**决策：**
- 全部 ✅ → 进入分类
- 有 ❌ → 打回发现者补充
- 重复 → 合并到现有卡片（见 §5.3）

### Phase 3：分类（Classification）

根据评估结果，确定：

| 决策 | 后续操作 |
|------|---------|
| 新卡片 | 分配类型（概念/方法/案例）+ 序号 |
| 合并到现有 | 找到目标卡片，追加内容 |
| 拒绝 | 记录原因，关闭申请 |

**序号分配：**
```bash
# 概念：KB_CAR_概念_001, KB_CAR_概念_002, ...
# 方法：KB_CAR_方法_001, KB_CAR_方法_002, ...
# 案例：KB_CAR_案例_001, KB_CAR_案例_002, ...
```

### Phase 4：编制（Authoring）

**执行者：** 入库执行者

1. 按 `卡片模板.md` 创建卡片
2. 填写 frontmatter：
   - `type`：概念/方法/案例
   - `code`：分配的标准编码
   - `version: 1.0.0`
   - `status: draft`（评审前）
3. 内容结构：
   - `## 摘要`：一句话（≤50字）
   - `## 来源`：原文出处或产生背景
   - `## 详情`：详细内容
   - `## 核心要点`：3-5条
   - `## 相关`：现有卡片 wikilink（如有）
4. 如有多个来源 → 在 `## 来源` 中全部列出

### Phase 5：评审（Review）

**执行者：** 评估者（可与 Phase 2 合并）

| 检查项 | 标准 |
|--------|------|
| 格式 | 符合卡片模板 frontmatter + 结构 |
| 重复 | 无与现有卡片内容重复（>30%相似度需合并） |
| 引用 | 相关链接正确 |
| 敏感 | 无敏感信息 |

**决策：**
- 通过 → 批准发布
- 需修改 → 注明原因，退回编制者

### Phase 6：发布（Publication）

**执行者：** 入库执行者

1. 将 frontmatter `status` 从 `draft` 改为 `active`
2. 更新 `system/受控文件清单.md`，新增条目
3. 更新 `system/SOP索引.md`，新增 SOP 条目（如是新 SOP）
4. 在 `daily/YYYY-MM-DD.md` 中记录知识入库

### Phase 7：归档（Archival）

**说明：** 此阶段针对来源文档本身（非卡片）

- 外部文档 URL → 存入 `inbox/来源记录.md`
- 内部产生知识 → 无需归档来源
- 入库申请记录 → 移入 `inbox/processed/`

## 6. 知识合并规则

当识别到与现有卡片重复时：

1. 打开目标卡片
2. 在 `## 相关` 中添加新来源链接
3. 如新内容丰富 → 追加到 `## 详情`
4. 更新 `updated` 日期
5. minor 版本号 +0.1
6. 在原卡片变更记录中登记

## 7. 记录要求

| 记录 | 保存位置 | 保存期限 |
|------|---------|---------|
| 入库申请表 | `inbox/YYYY-MM-DD_知识入库.md` | 1年 |
| 来源文档 | `inbox/来源记录.md` | 1年 |
| 发布的卡片 | `cards/{类型}/` | 永久 |
