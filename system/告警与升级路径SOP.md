# 告警与升级路径 SOP

> 编码：KB_SOP_007_v1.0.0 | 版本：1.0.0 | 2026-06-27 | 状态：active
> 监管要求：IATF16949:2016 §7.5 / §10.2 不合格与纠正措施
> 适用范围：所有 OpenClaw 生态内的异常、故障、告警响应

## 1. 目的

建立清晰的告警分级标准和升级路径，确保：
- 不同严重程度的异常有对应的响应级别
- 紧急情况能被快速识别并上报
- 升级路径明确，无人负责的死角
- 所有告警事件有记录、可追溯

## 2. 告警分级

| 级别 | 名称 | 定义 | 响应时间 | 处理者 |
|------|------|------|---------|--------|
| **P0** | 紧急 | 系统完全不可用、严重数据泄露 | 立即（5分钟内） | CEO 立即处理 |
| **P1** | 高 | 核心功能受损、外部通信中断 | 30 分钟内 | CEO + 相关域负责人 |
| **P2** | 中 | 功能降级、非核心服务异常 | 2 小时内 | 域负责人 |
| **P3** | 低 | 轻微问题、不影响使用 | 24 小时内 | 相关 Agent |
| **P4** | 提示 | 预防性提示、监控数据异常 | 72 小时内 | 相关 Agent |

## 3. 告警来源

| 来源 | 触发条件 | 对应级别 |
|------|---------|---------|
| Cron 健康检查失败 | `sys_health_daily` 报告 ❌ | P1-P2（视严重程度） |
| Feishu 发送失败 | `failureAlert` 触发 | P2 |
| API 限流/超载 | MiniMax 返回 429 | P2 |
| Session 异常 | session 无故断开 | P3 |
| 磁盘空间不足 | `< 1GB` | P2 |
| 知识库文件损坏 | frontmatter 丢失 | P3 |
| Skill 加载失败 | SKILL.md 格式错误 | P3 |
| 敏感信息泄露 | grep 扫描发现 | P1（立即处理） |

## 4. 升级路径

### 4.1 P0 紧急升级

```
发现者（任何Agent）
  │
  ▼ 立即通知
CEO（主会话 + 飞书）
  │
  ├── 成立响应小组（必要时）
  ├── 隔离问题（止血）
  ├── 分析根因
  └── 发布事件报告
```

### 4.2 P1 高优先升级

```
发现者（任何Agent）
  │
  ▼ 30分钟内
域负责人（如 CTO → coo 等）
  │
  ├── 分析问题
  ├── 制定修复方案
  └── 评估是否升级 P0
          │
          ├── 否 → 修复 → 验证 → 记录
          └── 是 → 升级 CEO
```

### 4.3 P2 中优先升级

```
发现者（任何Agent）
  │
  ▼ 2小时内
相关域负责人
  │
  ├── 确认问题
  ├── 安排修复计划
  └── 在 `inbox/` 开缺陷单
```

### 4.4 P3/P4 常规处理

```
发现者
  │
  ▼ 正常工作时间
在 `inbox/` 创建待办项
  │
  定期批量处理（每日 heartbeat 时）
```

## 5. 告警响应流程

### Phase 1：接收告警

| 步骤 | 操作 |
|------|------|
| 1.1 | 确认告警来源和内容 |
| 1.2 | 判定告警级别（P0-P4） |
| 1.3 | 记录告警接收时间 |
| 1.4 | 如为 P0/P1 → 立即启动升级 |

### Phase 2：评估

| 步骤 | 操作 |
|------|------|
| 2.1 | 分析告警数据/日志 |
| 2.2 | 确认影响范围 |
| 2.3 | 判断是否需要立即隔离 |
| 2.4 | 确认是否升级（P0↔P1 临界判断） |

### Phase 3：响应

| 级别 | 响应操作 |
|------|---------|
| P0 | 立即处理，暂停其他工作 |
| P1 | 30 分钟内给出初步分析 + 修复计划 |
| P2 | 2 小时内给出分析 + 修复时间表 |
| P3 | 24 小时内安排修复 |
| P4 | 72 小时内处理或关闭（评估风险） |

### Phase 4：修复

| 步骤 | 操作 |
|------|------|
| 4.1 | 执行修复（如需要变更文件 → 走文件变更 SOP） |
| 4.2 | 验证修复有效 |
| 4.3 | 确认无副作用 |
| 4.4 | 如涉及多个系统 → 通知相关方 |

### Phase 5：复盘

| 步骤 | 操作 |
|------|------|
| 5.1 | 编写事件报告（见 §8 模板） |
| 5.2 | 识别根本原因（RCA） |
| 5.3 | 提出预防措施（PA） |
| 5.4 | 更新 `inbox/事件报告_YYYY-MM-DD.md` |
| 5.5 | 将预防措施加入每日工作 |

## 6. 禁止事项

- **禁止**在 P0 时不升级而自行处理超过 10 分钟
- **禁止**在 P1 时超过 30 分钟无任何响应
- **禁止**隐瞒告警或降低告警级别
- **禁止**修复后不写事件报告
- **禁止**重复发生同一 P2+ 问题超过 2 次（须 RCA）

## 7. 记录要求

| 记录 | 保存位置 | 保存期限 |
|------|---------|---------|
| P0/P1 事件报告 | `daily/事件报告_YYYY-MM-DD.md` | 永久 |
| P2 事件摘要 | `daily/YYYY-MM-DD.md` | 永久 |
| P3/P4 待办 | `inbox/` | 处理后 30 天 |

## 8. 事件报告模板

```markdown
# 事件报告：{事件名称}

> 事件编号：INC-{YYYYMMDD}-{序号}
> 级别：P0/P1/P2
> 发现时间：YYYY-MM-DD HH:MM
> 恢复时间：YYYY-MM-DD HH:MM
> 影响时长：X 小时 X 分钟

## 事件概述
<!-- 一句话描述 -->

## 影响范围
<!-- 哪些系统/用户受影响 -->

## 时间线
| 时间 | 事件 |
|------|------|
| HH:MM | 发现 |
| HH:MM | 升级 |
| HH:MM | 开始修复 |
| HH:MM | 恢复 |

## 根因分析（RCA）
<!-- 为什么会发生？ -->

## 修复措施
<!-- 做了什么来修复？ -->

## 预防措施（PA）
<!-- 如何避免再次发生？ -->

## 改进建议
<!-- 更广泛的优化建议 -->
```

## 9. 升级决策矩阵

| 情况 | 升级决策 |
|------|---------|
| 系统完全不可用 | P0，立即升级 CEO |
| 影响所有用户 | P0/P1，取决于持续时间 |
| 影响部分用户 | P1/P2 |
| 外部服务中断（Feishu/MiniMax） | P1 |
| 敏感数据泄露风险 | P0，立即升级 CEO |
| 重复告警（同一问题 2次/24h） | 立即升级 |
| 未知原因告警 | P2 起步，可升级 |
