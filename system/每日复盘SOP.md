# 每日复盘 SOP

> 编码：KB_SOP_005_v1.0.0 | 版本：1.0.0 | 2026-06-27 | 状态：active
> 监管要求：IATF16949:2016 §7.5 成文信息控制
> 适用范围：CEO 及所有 Agent 的每日复盘执行

## 1. 目的

建立每日复盘的标准化流程，确保：
- 每日工作有记录、可追溯
- 问题能及时识别并进入升级流程
- 洞察能沉淀到知识库
- 为周/月/季复盘提供数据基础

## 2. 复盘时机

| 执行者 | 推荐时间 | 最晚时间 | 说明 |
|--------|---------|---------|------|
| CEO | 21:30-22:00 | 23:00 | 每日结束前 |
| Agent（主session） | 任务完成时 | 当日 23:00 | 主任务循环结束 |
| Agent（子session） | sessions结束前 | 每次会话结束 | 自动记录 |

## 3. 复盘流程

```
触发（定时/任务结束） → 打开模板 → 四维度填写 → 保存 → 自进化判断
```

### Phase 1：触发

**触发条件（满足任一）：**
- 每日 21:30（CEO 自动触发）
- 重大任务完成时
- 收到 HEARTBEAT 告警后

### Phase 2：填写四维度

使用 `templates/每日笔记模板.md`（KB_WI_001），按顺序填写：

| 维度 | 问题 | 要求 |
|------|------|------|
| **系统状态** | 当日系统是否正常？ | 简述（1-3句） |
| **今日完成** | 今日完成了什么？ | ≤3 条，每条一句话 |
| **问题与障碍** | 遇到了什么障碍？ | 有则记录，无则写"无" |
| **洞察与想法** | 有什么新认识？ | 有则记录，无则写"无" |
| **明日计划** | 明天最重要的1-2件事 | 1-2 件，优先级排序 |

**填写规范：**
- 每条内容不超过 2 行
- 使用 `[[卡片名]]` 引用相关知识库卡片
- 问题如需跟进 → 写入 `inbox/问题单_YYYY-MM-DD.md`

### Phase 3：保存

1. 文件保存到 `daily/YYYY-MM-DD.md`
2. frontmatter 由模板自动生成
3. footer 自动追加

### Phase 4：自进化判断

**触发条件（满足任一）：**
- 同一问题连续出现 2 天
- 某任务耗时环比增长 >30%
- 发现新的有效方法

**处理流程：**
1. 写入 `inbox/问题单_YYYY-MM-DD.md`
2. 如判断为进化需求 → 触发 `KB_SOP_008`（自进化体系闭环）
3. 不满足进化条件 → 常规记录，结束

## 4. 复盘输出

| 输出 | 保存位置 | 用途 |
|------|---------|------|
| 每日笔记 | `daily/YYYY-MM-DD.md` | 永久记录 |
| 问题单（如有） | `inbox/问题单_YYYY-MM-DD.md` | 问题跟踪 |
| 进化需求（如有） | 触发 KB_SOP_008 | 知识库更新 |

## 5. 禁止事项

- **禁止**将复盘写成流水账（重在洞察，非记录）
- **禁止**超过 3 条"今日完成"（聚焦重点）
- **禁止**在复盘中直接处理问题（先记录，次日处理）
- **禁止**跳过"明日计划"（无计划=无方向）

## 6. 复盘质量标准

| 检查项 | 标准 |
|--------|------|
| 完整性 | 5 个维度全部填写 |
| 简洁性 | 每条 ≤2 行 |
| 可操作性 | "明日计划"可执行 |
| 可追溯性 | 引用相关知识库卡片 |

## 7. 自进化集成

每日复盘是自进化系统「感知层」的主要数据来源：

```
每日复盘 → 每日夜间 Dreaming System 整合
         → 识别模式 → 生成洞察
         → 触发知识入库（KB_SOP_006）
         → 更新 SOP（KB_SOP_003）
         → 执行层反馈
```
