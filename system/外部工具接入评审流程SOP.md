# 外部工具接入评审流程 SOP

> 编码：QL_SOP_003_v1.0.0 | 版本：1.0.0 | 2026-06-27 | 状态：active
> 监管要求：IATF16949:2016 §7.5 成文信息控制
> 适用范围：所有新外部工具/服务接入 OpenClaw 生态前的安全评审

## 1. 目的

确保每个外部工具接入都经过充分的安全和可行性评估，防止：
- 凭证泄露风险
- 数据外泄风险
- 单点依赖风险
- 成本失控风险

## 2. 外部工具分类

| 类别 | 风险等级 | 示例 |
|------|---------|------|
| 消息通道 | 高 | Feishu、Slack |
| AI 能力 | 中 | MiniMax、OpenAI |
| 开发辅助 | 中 | Codex++ |
| 存储/文件 | 高 | AWS S3、SMB |
| Webhook | 高 | 外部回调 |
| 数据服务 | 中 | 数据库、向量库 |

## 3. 评审流程

```
需求申请 ──→ 安全评估 ──→ 可行性评审 ──→ 审批 ──→ 实施 ──→ 上线确认
```

### Phase 1：需求申请

**申请者：** 任何 Agent 或用户

| 步骤 | 操作 |
|------|------|
| 1.1 | 在 `inbox/` 创建 `外部工具接入申请_YYYY-MM-DD.md` |
| 1.2 | 填写：工具名称/用途/提供者/预计用量 |
| 1.3 | 提交给 CEO |

**申请内容：**

```markdown
## 外部工具接入申请

> 申请日期：YYYY-MM-DD
> 申请者：{who}

### 工具基本信息
- 工具名称：{name}
- 提供者：{provider}
- 用途：{purpose}

### 数据流向
<!-- 数据是从 OpenClaw 出去，还是外部数据进来 -->

### 凭证需求
- 需要哪些凭证（API Key/Token/SSH Key）？
- 凭证是否可 revocation？

### 预计用量
- 预计调用频率：
- 预计成本：
```

### Phase 2：安全评估

**评估者：** CEO（可委托 qm 执行）

| 评估项 | 标准 | 判定 |
|--------|------|------|
| 凭证安全性 | 凭证是否可 revocation | ✅/❌ |
| 数据隔离 | 数据是否离开 OpenClaw 环境 | 明确 |
| 访问控制 | 是否有 IP 白名单或权限控制 | ✅/❌ |
| 日志能力 | 是否支持操作日志 | ✅/❌ |
| 故障恢复 | 失败时是否有自动重试 | ✅/❌ |
| 合规性 | 是否符合 QL_SOP_002（敏感信息）| ✅/❌ |

**P0 项（必须全部通过）：**
- 凭证可 revocation
- 不在群聊中传输凭证

**决策：**
- 全部 P0 通过 → 进入可行性评审
- 有 P0 未通过 → 否决申请

### Phase 3：可行性评审

**评审者：** CTO（技术可行性）+ COO（成本/资源）

| 维度 | 评审内容 | 判定 |
|------|---------|------|
| 技术可行性 | 是否能与 OpenClaw 集成？ | ✅/⚠️/❌ |
| 成本评估 | 成本是否在预算内？ | ✅/❌ |
| 依赖性 | 是否为单点依赖？有备用方案？ | ✅/⚠️/❌ |
| 维护成本 | 集成后的维护负担是否可控？ | ✅/⚠️/❌ |

**决策：**
- 全部 ✅ → 报 CEO 审批
- 有 ⚠️ → 制定缓解方案后报 CEO
- 有 ❌ → 否决申请

### Phase 4：审批

**审批者：** CEO

| 工具风险等级 | 审批要求 |
|-------------|---------|
| 高风险 | CEO 单独审批 |
| 中风险 | CEO + 相关域负责人 |
| 低风险 | CEO 或委托 COO |

### Phase 5：实施

**执行者：** 相关域 Agent

按 `QL_SOP_001`（外部工具集成规范）执行：
1. 凭证配置（环境变量）
2. 调用代码实现
3. 错误处理实现
4. 日志记录实现

### Phase 6：上线确认

**评审者：** CEO

| 检查项 | 判定 |
|--------|------|
| 凭证不在任何 .md 文件中 | ✅/❌ |
| 有日志记录 | ✅/❌ |
| 有降级方案 | ✅/❌ |
| 已更新 `TOOLS.md` | ✅/❌ |

## 4. 快速评审（低风险工具）

对于风险等级为"低"的工具，简化流程：

```
需求申请 → CEO 快速审批（≤24h） → 实施 → 上线确认
```

跳过 Phase 2（安全评估）和 Phase 3（可行性评审）。

## 5. 凭证管理规范

所有接入工具的凭证必须满足：

| 要求 | 说明 |
|------|------|
| 不明文存储 | 凭证不得出现在 .md 文件中 |
| 环境变量优先 | 凭证存储在环境变量 |
| 最小权限 | 凭证权限 = 最小所需权限 |
| 可 revocation | 凭证可随时撤销 |

## 6. 记录要求

| 记录 | 保存位置 | 保存期限 |
|------|---------|---------|
| 接入申请 | `inbox/外部工具接入申请_YYYY-MM-DD.md` | 1 年 |
| 评审报告 | `inbox/外部工具接入申请_YYYY-MM-DD.md`（合并）| 永久 |
| 上线确认 | `daily/YYYY-MM-DD.md` | 永久 |
| 凭证配置记录 | `TOOLS.md` | 永久 |

## 7. 禁止事项

- **禁止**在未经评审的情况下接入高风险外部工具
- **禁止**在群聊中讨论或发送凭证
- **禁止**在 .md 文件中记录真实凭证
- **禁止**使用无法 revocation 的凭证
