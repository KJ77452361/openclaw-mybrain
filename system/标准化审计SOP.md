# 标准化审计 SOP

> 编码：KB_SOP_004_v1.0.0 | 版本：1.0.0 | 2026-06-27 | 状态：active
> 监管要求：IATF16949:2016 §7.5 成文信息控制
> 适用范围：所有 Agent 工作区、所有受控文件的合规性审计

## 1. 目的

建立标准化的审计流程，对 OpenClaw 全生态的受控文件进行定期合规检查，确保：
- 所有 Agent 工作区符合标准化要求
- 所有受控文件（卡片/SOP/模板）符合格式规范
- 敏感信息泄露风险持续可控
- 审计结果可追溯、可复审

## 2. 适用范围

| 审计对象 | 范围 |
|---------|------|
| Agent 工作区 | `agents/` 下 17 个 + `workspace/agents/` 下 4 个 |
| 受控文件 | MyBrain 内所有 active 文件 |
| Skills | `skills/` 下所有 SKILL.md |
| Cron 定义 | `cron/definitions/` 下所有定义文件 |

## 3. 职责

| 角色 | 职责 |
|------|------|
| 主审（CEO） | 制定审计计划、执行审计、发布审计报告 |
| 协审（知识库管理员） | 辅助文件检查、记录整理 |
| 被审计方（各 Agent） | 配合提供文件、接收整改通知 |

## 4. 审计标准

### 4.1 Agent 工作区必须文件（5个）

| 文件 | 检查内容 |
|------|---------|
| `AGENTS.md` | 存在、有角色定义、有行为准则 |
| `SOUL.md` | 存在、有核心人格描述 |
| `IDENTITY.md` | 存在、有版本和更新日期 |
| `USER.md` | 存在、用户信息完整 |
| `MEMORY.md` | 存在、长期记忆内容存在 |

### 4.2 受控文件格式检查

| 检查项 | 标准 |
|--------|------|
| frontmatter | 第一行为 `---`，含 title/type/version |
| 版本 footer | 最后一行以 `*version` 开头 |
| 敏感信息 | 不含 `sk-` 明文、Bearer token、密码 |
| wikilink | 使用 `[[卡片名]]` 格式引用 |
| 变更记录 | 文件底部有变更记录表 |

### 4.3 Skill 格式检查

| 检查项 | 标准 |
|--------|------|
| 文件位置 | `skills/{name}/SKILL.md` |
| frontmatter | 含 `name`/`description`/`version` |
| 描述质量 | 有描述、前置条件、使用示例 |

## 5. 审计流程

```
计划 ──→ 执行 ──→ 记录 ──→ 判定 ──→ 整改 ──→ 关闭
```

### Phase 1：审计计划

| 步骤 | 操作 | 时间 |
|------|------|------|
| 1.1 | 制定审计范围（全面/专项） | 审计前 1 天 |
| 1.2 | 确认审计标准和判定规则 | 审计前 1 天 |
| 1.3 | 通知被审计方准备 | 审计前 1 天 |
| 1.4 | 准备检查清单（见 §6） | 审计前 1 天 |

### Phase 2：执行审计

| 步骤 | 操作 |
|------|------|
| 2.1 | 文件存在性检查（对照清单） |
| 2.2 | frontmatter 格式检查 |
| 2.3 | 版本 footer 检查 |
| 2.4 | 敏感信息扫描（grep 关键字） |
| 2.5 | 内容一致性抽查（跨文件引用） |
| 2.6 | 记录每项检查结果（通过/失败/不适用） |

### Phase 3：结果记录

| 步骤 | 操作 |
|------|------|
| 3.1 | 汇总所有失败项 |
| 3.2 | 填写审计报告（见 `daily/审计报告_YYYY-MM-DD.md`） |
| 3.3 | 判定审计结论（通过/条件通过/不通过） |

**审计结论判定：**
- **通过**：0 个不合规项
- **条件通过**：minor 不合规项 ≤3 个，可整改后复审
- **不通过**：major 不合规项 ≥1 个，或 minor 不合规项 >3 个

### Phase 4：整改跟踪

| 步骤 | 操作 | 期限 |
|------|------|------|
| 4.1 | 发布审计报告给被审计方 | 审计当天 |
| 4.2 | 被审计方提交整改计划 | 1 天内 |
| 4.3 | 执行整改 | minor 3天内 / major 7天内 |
| 4.4 | 整改验收 | 整改完成后 1 天内 |
| 4.5 | 关闭审计，更新状态 | 验收通过后 |

### Phase 5：审计关闭

| 步骤 | 操作 |
|------|------|
| 5.1 | 确认所有失败项已整改或接受风险 |
| 5.2 | 更新 `SOP索引.md` 中审计状态 |
| 5.3 | 将审计报告存入 `daily/审计报告_YYYY-MM-DD.md` |

## 6. 审计检查清单（自动化脚本）

```bash
# 文件存在性
for f in AGENTS.md SOUL.md IDENTITY.md USER.md MEMORY.md; do
  [ -f "$f" ] && echo "✅ $f" || echo "❌ $f 缺失"
done

# frontmatter 检查
head -1 file.md | grep -q "^---$" && echo "✅ frontmatter" || echo "❌ frontmatter"

# 版本 footer 检查
tail -1 file.md | grep -q "^\*version" && echo "✅ footer" || echo "❌ footer"

# 敏感信息扫描
grep -E "sk-[a-zA-Z0-9]{20,}|Bearer [a-zA-Z0-9]{20,}" file.md && echo "❌ 敏感信息"
```

## 7. 审计频次

| 类型 | 频率 | 执行时间 | 执行者 |
|------|------|---------|--------|
| 全面审计 | 每季度一次 | 季度首月 | CEO |
| Agent 工作区审计 | 每月一次 | 每月 25 日 | CEO |
| 专项审计（敏感信息） | 每周一次 | 每周一 | 自动化脚本 |
| Cron 定义审计 | 每季度一次 | 季度首月 | CEO |
