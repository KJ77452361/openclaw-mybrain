# 文件版本管理规范

> 编码：KB_CAR_方法_003_v1.0.0 | 版本：1.0.0 | 2026-06-27 | 状态：active
> 监管要求：IATF16949:2016 §7.5 成文信息控制
> 详细流程见：`system/文件变更控制SOP.md`（KB_SOP_003_v1.0.0）

## 摘要

规范 OpenClaw 生态内所有受控文件的版本编号规则和变更分级。

## 版本号格式

```
v主版本.次版本.修订号
```

**示例：** `v1.0.0` → `v1.1.0` → `v2.0.0`

## 变更分级

| 级别 | 触发条件 | 版本变更 | 说明 |
|------|---------|---------|------|
| **patch** | 错别字、格式调整、注释补充 | `v{x}.{y}.{z+1}` | 不改变实质内容 |
| **minor** | 内容变化 <20%、新增小节、补充案例 | `v{x}.{y+1}.{0}` | 实质性变化但非重构 |
| **major** | 重写、删除章节、内容变化 ≥20%、新机制引入 | `v{x+1}.{0}.{0}` | 结构性重大变更 |

## 版本状态定义

| 状态 | 含义 | 操作 |
|------|------|------|
| **active** | 正式使用，所有人可执行 | 当前有效版本 |
| **draft** | 草稿，评审中 | 不可执行，需评审 |
| **obsolete** | 已作废 | 移入 `memory/archive/`，禁止使用 |

## 文件标识规则

每个受控文件 frontmatter 必须包含：

```yaml
---
version: 1.0.0        # 当前版本号
status: active        # active / draft / obsolete
updated: YYYY-MM-DD   # 最后更新日期
---
```

## 变更记录要求

每个受控文件底部必须包含变更记录表：

```markdown
## 变更记录

| 日期 | 版本 | 变更内容 | 变更人 |
|------|------|---------|--------|
| YYYY-MM-DD | v1.0.0 | 首版发布 | King |
```

## 作废文件处理

1. 将文件 frontmatter 中 `status` 改为 `obsolete`
2. 在文件名前加 `_obsolete_{日期}_` 前缀
3. 将文件移入 `memory/archive/YYYY-MM/`
4. 更新 `system/受控文件清单.md` 中的状态
